5. BASE URL AND AUTH RULE

Backend base URL:
- production: https://api.anushatechnologies.com
- local example: http://localhost:9000

JWT header:
- Authorization: Bearer <jwtToken>

Important backend note:
- There is no dedicated refresh-token API for delivery app users
- only admin has refresh endpoint
- for delivery app, when JWT expires, app should:
  1. get fresh Firebase ID token
  2. call POST /api/delivery/auth/login again
  3. save new JWT
  4. continue app session

6.1 Existing Delivery Partner Login Flow

Step 1:
- open Login screen
- enter 10-digit phone number
- call GET /api/delivery/auth/check-phone/{phone}

Step 2:
- if exists = true, send Firebase OTP
- move to OTP screen

Step 3:
- verify OTP in Firebase
- get Firebase ID token

Step 4:
- call POST /api/delivery/auth/login
- body:
  {
    "firebaseIdToken": "<firebase-id-token>",
    "fcmToken": "<device-fcm-token>"
  }

Step 5:
- save jwtToken in AsyncStorage
- call GET /api/delivery-app/status

Step 6 route decision:
- if approvalStatus = APPROVED -> go to /(tabs)
- if approvalStatus = PENDING or REJECTED -> go to /verification


6.2 New Delivery Partner Signup Flow

Step 1:
- enter phone number in register flow
- call GET /api/delivery/auth/check-phone/{phone}

Step 2:
- if not exists, send Firebase OTP
- verify OTP
- get Firebase ID token

Step 3:
- collect:
  - firstName
  - lastName
  - vehicleType
  - vehicleModel
  - registrationNumber
  - profile photo

Step 4:
- upload profile photo first
- call POST /api/delivery/auth/upload-profile-photo
- multipart:
  - file

Step 5:
- call POST /api/delivery/auth/signup
- body:
  {
    "firebaseIdToken": "<firebase-id-token>",
    "firstName": "Ravi",
    "lastName": "Kumar",
    "vehicleType": "BIKE",
    "vehicleModel": "Honda Shine",
    "registrationNumber": "TS09AB1234",
    "profilePhotoUrl": "<uploaded-photo-url>",
    "fcmToken": "<device-fcm-token>"
  }

Step 6:
- save jwtToken
- save deliveryPersonId

Step 7:
- upload required KYC documents one by one
- call POST /api/documents/upload

Step 8:
- route to /verification
- keep checking GET /api/delivery-app/status

Step 9:
- when admin approves -> go to /(tabs)


6.3 Daily Partner App Flow

Step 1:
- open app
- if jwt exists, call GET /api/delivery-app/status

Step 2:
- if valid token and user exists -> restore user session
- if 401 -> re-login using Firebase ID token

Step 3:
- home screen shows:
  - online/offline status
  - total earnings
  - active orders count
  - available orders badge

Step 4:
- rider goes online
- call PUT /api/delivery-app/online-status

Step 5:
- app polls available order list
- call GET /api/delivery-app/available-orders

Step 6:
- rider accepts or rejects order

Step 7:
- accepted order goes to Orders screen

Step 8:
- rider reaches store
- verifies pickup OTP
- starts live location updates
- delivers order

Step 9:
- order moves to completed
- earnings and payouts update

9.3 Validation Rules

Use GET /api/documents/validation-rules to show help text.

Local validation to keep in app:
- Aadhaar: 12 digits
- PAN: ABCDE1234F
- Driving License: valid alphanumeric
- RC number: valid vehicle registration format
- Insurance: optional, if file uploaded then number required


9.4 Document Upload API

POST /api/documents/upload

multipart/form-data:
- deliveryPersonId
- documentType
- documentNumber
- file

Example:
- deliveryPersonId: 5
- documentType: PAN_CARD
- documentNumber: ABCDE1234F
- file: <image>


9.5 Document Status Handling In App

Each document should show one status chip:
- NOT_UPLOADED
- PENDING
- APPROVED
- REJECTED
- NEEDS_REUPLOAD

Rule:
- APPROVED -> locked
- PENDING -> locked
- REJECTED or NEEDS_REUPLOAD -> allow reupload


9.6 Important Backend Note

Current backend placeholder endpoints:
- GET /api/documents/{documentId}
- DELETE /api/documents/{documentId}

These are not fully implemented yet.

So in app:
- do not depend heavily on delete flow
- primary flow should be upload and reupload
- status should come from GET /api/delivery-app/status

11. CLEAN TECHNICAL IMPLEMENTATION RULES

11.1 Use JWT-based endpoints first

Preferred for app:
- /api/delivery-app/*

Use id-based endpoints only when needed for:
- statistics
- active/completed list
- fallback lookups


11.2 One Notification System Only

Current app mixes:
- Firebase messaging
- Expo notification token style

Clean plan:
- use Firebase FCM only
- save FCM token using POST /api/save-token
- remove Expo push token dependency for delivery app production flow


11.3 One Session Recovery Strategy

Do not build fake refresh token flow.

On 401:
1. try Firebase current user getIdToken(true)
2. call POST /api/delivery/auth/login again
3. replace old JWT
4. retry failed request
5. if this fails -> logout and go to login


11.4 Remove Placeholder Data

Must remove from current screens:
- hardcoded OTP values
- hardcoded map coordinates
- fake payout success
- fake order amounts
- fake vendor/customer strings
- fake "today earnings" numbers
